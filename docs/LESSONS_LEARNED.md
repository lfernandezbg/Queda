# Queda — Lessons Learned

| Context | Incorrect Assumption | Evidence | Final Solution | Rule to Apply |
|---|---|---|---|---|
| E2E Barcode Scanning | CI/Maestro can use virtual camera frames. | GitHub Actions emulators often lack reliable virtual camera/sensor support. | Use internal intent hooks to inject scan events in E2E builds. | E2E tests for hardware-dependent features must use deterministic software hooks. |
| Room Migration Tests | `MigrationTestHelper` constructor is static across versions. | Room 2.4.0+ changed constructor and requires schema JSONs in assets. | Include schemas in `androidTest` assets and use the updated constructor. | Always verify Room testing requirements when upgrading dependencies. |
| Material3 Modals | Modal content test tags are always reachable from parent window. | Maestro failure artifacts showed modal descendants invisible when tag was on wrapper. | Apply `testTagsAsResourceId = true` explicitly inside the modal content window. | Expose semantics specifically within separate windows (Modals/Dialogs). |
| ML Kit Lifecycle | ML Kit clients can be created per frame in the analyzer. | Performance degraded and memory leaked in instrumented tests. | Create one `BarcodeScanner` client per scanner screen lifecycle and close it explicitly. | Never create ML Kit clients per camera frame; use explicit lifecycle ownership. |
| CameraX Cleanup | `unbindAll()` in `onDispose` is sufficient. | Executor and Analyzer persisted after leaving the screen in debug traces. | Shutdown the executor and close the analyzer explicitly in `DisposableEffect`. | CameraX analyzer, executor and scanner require explicit lifecycle ownership. |
| E2E Isolation | Deep links in `AndroidManifest.xml` are build-type neutral. | `verify-release-isolation.ps1` failed when `queda-e2e` scheme leaked into main manifest. | Move E2E intent filters and activities to the `e2e` source set manifest. | E2E schemes must not exist in release source. |
| Barcode Lookup | Nullable return can represent both 'not found' and 'error'. | ViewModel couldn't distinguish between a new product and a database failure. | Return an explicit sealed result (Found, NotFound, StorageFailure). | Nullable lookup cannot represent storage failure. |
| Room Uniqueness | Generic `StorageFailure` is enough for uniqueness conflicts. | Users saw a generic error when adding a product with an existing barcode. | Detect existing barcodes explicitly and return a specific `DuplicateBarcode` result. | Database uniqueness errors require explicit domain mapping. |
| Migration Assertions | `assert(condition)` is sufficient for JVM tests. | Kotlin's `assert` is often ignored at runtime unless `-ea` is passed. | Use JUnit's `assertEquals`, `assertTrue`, etc. | Migration tests must use real JUnit assertions. |
| Maestro Selectors | Resource IDs are always reliable for tapping. | `quantity_action_confirm` was unreachable in some Material3 modal versions. | Use visible button text (e.g., `tapOn: "Consumir"`) for critical actions. | Material3 modal resource IDs are not reliable for Maestro. |
| E2E Isolation | Generic names like "initialBarcode" are safe for production code. | A test hook can leak into release even without an "e2e" name if it's in a production file. | Remove all scan-specific E2E parameters and Intent extras from production Activity/Composable. | Production Activity and Composable must not accept E2E-only parameters. |
| E2E Intent Extras | Intent extras used only by E2E are harmless production backdoors. | Security audit flagged raw scan extra as a potential production backdoor. | Implement E2E orchestration entirely within the `e2e` source set using its own Activity and host. | E2E orchestration must remain outside of the production Activity. |
| Maestro Timing | 10s is always enough for local operations. | Maestro screenshots showed state arrived immediately after timeout. | Use bounded waits (e.g., 30s) based on demonstrated asynchronous behavior. | Maestro screenshots are evidence for timeout adjustment, not just failure. |
| Lifecycle Observers | Capturing a Compose State in `DisposableEffect` is safe. | Lifecycle observers often capture stale state when using `uiState` directly. | Use `rememberUpdatedState` for values used inside lifecycle observers. | Lifecycle observers must not capture stale Compose state. |
| ViewModel Concurrency | Checking `StateFlow` before launching a coroutine is atomic. | Two rapid calls can both pass the check before the first launch updates the state. | Use an atomic gate (e.g., `AtomicBoolean`) synchronously before launching. | Checking StateFlow before launching is not an atomic concurrency gate. |
